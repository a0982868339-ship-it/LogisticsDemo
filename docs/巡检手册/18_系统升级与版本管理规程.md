## 1. 版本管理策略

### 1.1 版本号规范

EMIS系统遵循**语义化版本（Semantic Versioning 2.0.0）**规范：  
格式：`MAJOR.MINOR.PATCH[-PRE_RELEASE][+BUILD_META]`

| 版本段 | 含义 | 示例触发场景 |
|---|---|---|
| MAJOR（主版本）| 不兼容的架构变更或API重大修改 | 更换AI框架；调度引擎重构 |
| MINOR（次版本）| 新增功能，向后兼容 | 新增巡检场景；新增传感器类型 |
| PATCH（补丁）| Bug修复或小参数调整 | 修复错误码映射；调整告警阈值 |
| PRE_RELEASE | 预发布版本（alpha/beta/rc）| 测试阶段 |

## 2. 升级流程

### 2.1 升级分类与审批

| 升级类型 | 定义 | 审批要求 | 升级窗口 |
|---|---|---|---|
| **紧急补丁（Hotfix）** | 修复生产环境严重Bug或安全漏洞 | IT主管+CISO（安全漏洞）两人审批 | 随时可发布，事后报管委会 |
| **定期补丁（Patch）** | 计划内小版本更新 | IT主管审批 | 每月第一个周日凌晨01:00~05:00 |
| **功能升级（Minor）** | 新功能或算法模型更新 | IT主管+智能运营中心主任联合审批 | 每季度次月第一个周日 |
| **大版本升级（Major）** | 架构级重构或重大变更 | 管理委员会全体审批，提前8周评审 | 特定窗口（一般年度1~2次）|
| **AI模型热更新** | 不涉及代码的模型文件替换 | AI算法组组长+合规审计部审批 | 每月第三个周日凌晨 |

### 2.2 标准升级流程（Minor以上）

```
阶段一：需求与计划（升级前8~12周）
├── 技术方案评审（Architecture Review Board，ARB）
├── 影响评估（对海关接口、业务连续性的影响）
├── 回滚方案设计（详细的回滚步骤和时间估算）
└── 通知受影响部门（合规审计部、安全组、仓储作业部）

阶段二：测试与验证（升级前4~6周）
├── 在【测试环境】完成功能测试（测试环境地址：test.syzbq.internal）
├── 在【压测环境】完成性能测试（目标：满负荷的120%）
├── 在【验收环境】（Staging）完成UAT验收
├── 合规审计部确认：升级不影响海关数据上报合规性
└── 安全扫描：SAST静态代码扫描 + DAST动态扫描（通过率100% for Critical）

阶段三：灰度发布（升级前1~2周）
├── 选取5%机器人（约18台）作为灰度批次，部署新固件/算法
├── 观察7天：KPI是否达标，錯误率是否在正常范围
└── 灰度通过 → 制定全量发布计划；灰度失败 → 回滚并重排计划

阶段四：全量发布（升级维护窗口）
├── T-30min：通知各部门，调度系统进入维护模式
├── T-15min：所有机器人完成当前任务，返回充电站（或停靠安全区）
├── T-0：开始滚动升级（按区域分批，15%→50%→100%）
├── 每批升级后验证（5分钟观察窗口）
├── 全量升级完成后，运行【升级验证测试套件】（自动化，约15分钟）
└── 通过后，退出维护模式，恢复正常调度

阶段五：升级后监控（升级后48小时）
├── 24小时核心指标异常监控（错误率×3标准差触发告警）
├── 与上次版本运营数据对比（KPI变化超±5%须评估）
└── 48小时后关闭回滚快速通道（超过48小时若需回滚须走完整流程）
```

### 2.3 机器人固件OTA升级流程

机器人支持**无感OTA**（On-The-Air）升级，升级在机器人充电期间进行：

| 参数 | 配置 |
|---|---|
| OTA服务器地址 | `ota.syzbq.internal:8080` |
| 固件包签名 | RSA-4096（制造商私钥签名，机器人公钥验证）|
| 包完整性校验 | SHA-256哈希 |
| 下载并发数 | 最多同时30台（控制带宽占用）|
| 升级触发条件 | 机器人充电中 + 电量>30% + 系统空闲 |
| 升级超时保护 | 单台升级>30分钟自动回滚 |
| 双分区设计 | 机器人内置A/B双系统分区，升级写入B区，失败自动Boot回A区 |

## 3. 变更控制

### 3.1 变更分类

| 变更类型 | 案例 | 影响评估要求 |
|---|---|---|
| 代码变更 | 新功能、Bug修复 | Git PR + Code Review（≥2人审批）|
| 配置变更 | 告警阈值、模型参数 | CMDB变更单（CHANGE-CFG-XXX）+ 审批 |
| 架构变更 | 新增服务器、网络变更 | ARB评审 + 上级审批 |
| 数据变更 | 禁限清单手动更新 | 合规审计部审批 |
| 证书/密钥变更 | SSL证书更新 | IT主管+CISO审批 |

### 3.2 紧急变更程序

当发生以下情况需要紧急变更：
- 生产环境出现数据丢失
- 海关接口中断超过1小时
- 安全漏洞被利用

**紧急变更步骤：**
1. 通知IT主管 + 相关技术负责人
2. 在CMDB系统创建紧急变更单（EMERGENCY-CHANGE），标明「紧急」
3. 快速评估变更：影响范围？回滚方案？预计恢复时间？
4. 双人操作（修改人+审核人同时在场）
5. 变更完成后，24小时内提交详细变更报告

## 4. 回滚程序

### 4.1 服务层回滚

**回滚命令（各服务使用Helm管理）：**

```bash
# 查看当前部署历史
helm history emis-dispatch -n emis-prod

# 回滚到上一版本
helm rollback emis-dispatch 0 -n emis-prod

# 回滚到指定版本（REVISION=5）
helm rollback emis-dispatch 5 -n emis-prod

# 验证回滚状态
kubectl rollout status deployment/emis-dispatch -n emis-prod
```

**回滚时间目标：**
- Tier-1服务回滚完成：≤10分钟
- Tier-2服务回滚完成：≤20分钟
- AI模型回滚（切换MLflow注册版本）：≤5分钟

### 4.2 机器人固件回滚

机器人固件回滚通过双分区自动完成。手动强制回滚：

```bash
# 通过OTA服务器强制回滚单台机器人（以X7P-128为例）
curl -X POST \
  https://ota.syzbq.internal:8080/api/rollback \
  -H "Authorization: Bearer {AdminToken}" \
  -d '{"robotId": "X7P-128", "targetVersion": "FW-3.3.2"}'
```

## 5. 版本相关错误码

| 错误码 | 描述 | 处置 |
|---|---|---|
| UPG-E001 | OTA固件包签名验证失败 | 检查签名密钥，禁止该包安装，联系制造商 |
| UPG-E002 | 固件升级超时（>30分钟）| 系统自动回滚，检查网络和OTA服务器状态 |
| UPG-E003 | 版本不兼容（机器人固件与调度引擎）| 按版本兼容矩阵（见附录A）降级或升级对应组件 |
| UPG-E004 | 灰度机器人错误率超阈值 | 终止灰度发布，全量保留旧版，排查原因 |
| UPG-E005 | Helm回滚失败 | 手动切换K8s部署版本；联系DevOps组 |
| UPG-E006 | 新版本AI模型精度下降>2% | 立即回滚模型，通知算法组重新验证 |
| UPG-E007 | 升级后海关接口报文格式不兼容 | 立即回滚接口服务，联系合规审计部评估影响 |
